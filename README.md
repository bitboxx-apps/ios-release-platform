# iOS Release Platform

> **[日本語版はこちら (README_ja.md)](README_ja.md)**

A GitHub Template that generates a **compilable Xcode project** and a **production-ready release pipeline** from environment variables alone.

Clone it, fill in `.env`, run one command — your app builds in CI with signed certificates and deploys to TestFlight.

---

## Why This Exists

Shipping an iOS app to the App Store requires coordinating Xcode signing, certificate management, CI pipelines, and App Store Connect credentials. Most teams bolt this together ad-hoc, producing fragile setups that break on team changes, certificate expiry, or CI migration.

This template treats release infrastructure as a first-class concern — not an afterthought.

**What you get:**

- A Xcode project generated from your `.env` values — correct bundle ID, team, display name, manual signing from the start
- fastlane match for zero-touch certificate management
- GitHub Actions that build, sign, and deploy to TestFlight
- A single bootstrap command that provisions everything
- Layered environment configuration with JSON Schema validation
- Enforced separation: product code in `app/`, infrastructure in `release/`

**What you don't get:**

- Interactive wizards or magic defaults
- Auto-installed system dependencies
- Implicit certificate rotation
- Anything that silently mutates state

## Quick Start

### 1. Create your repository

Click **"Use this template"** on GitHub to create a new repository.

### 2. Check your environment

```bash
./release/bootstrap/doctor.sh
```

```
[OK] Git: git version 2.43.0
[OK] GitHub CLI: gh version 2.40.0
[OK] Ruby: 3.3.0
[OK] Bundler: 2.5.3
[OK] fastlane: 2.219.0
[OK] Xcode: Xcode 15.2
[OK] All checks passed. Environment is ready.
```

Doctor validates prerequisites but never installs them.

### 3. Configure

```bash
cp .env.platform.example .env.platform
cp .env.app.example      .env.app
```

**`.env.platform`** — organization-level secrets:
```
APPLE_TEAM_ID=A1B2C3D4E5
MATCH_GIT_URL=git@github.com:your-org/ios-signing.git
MATCH_PASSWORD=your-encryption-passphrase
```

**`.env.app`** — your app's identity:
```
APP_NAME=MyApp
APP_DISPLAY_NAME=My App
APP_BUNDLE_ID=com.company.myapp
MATCH_TYPE=appstore
ASC_KEY_ID=XXXXXXXXXX
ASC_ISSUER_ID=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
ASC_KEY_PATH=~/keys/AuthKey_XXXXXXXXXX.p8
MATCH_GIT_PRIVATE_KEY_PATH=~/.ssh/match_key
```

`APP_NAME` is the Xcode identifier (no spaces). `APP_DISPLAY_NAME` is what appears on the iOS home screen (spaces, Japanese, emoji all OK).

### 4. Bootstrap

```bash
./release/bootstrap/bootstrap.sh --init
```

This single command:

1. Validates your configuration against the JSON Schema
2. **Generates the Xcode project** (`app/MyApp.xcodeproj`) with correct bundle ID, team, display name, and manual signing
3. Verifies Ruby >= 3.2 and fastlane
4. Generates fastlane configuration from templates
5. Configures match and generates signing certificates
6. Provisions GitHub Secrets (ASC key is base64-encoded, SSH key for match)

### 5. Open and develop

```bash
open app/MyApp.xcodeproj
```

The project is ready — bundle ID, team, display name, and signing are all configured. Start writing your app.

### 6. Commit and deploy

```bash
git add .env.app app/ release/platform/fastlane/
git commit -m "Initialize project"
git push origin main
```

Tag a version to trigger TestFlight deployment:

```bash
git tag v1.0.0
git push origin v1.0.0
```

## Repository Structure

```
.
├── app/                              # Product code — generated by bootstrap
│   ├── <APP_NAME>.xcodeproj/         #   Xcode project (manual signing, correct team/bundle ID)
│   └── <APP_NAME>/                   #   Swift source, assets
│
├── release/                          # Release infrastructure — never touches app/
│   ├── bootstrap/
│   │   ├── doctor.sh                 #   Environment validator
│   │   ├── bootstrap.sh              #   Primary initializer (macOS)
│   │   └── bootstrap.ps1             #   Secret provisioning (Windows)
│   ├── scripts/
│   │   ├── create_project.sh         #   Xcode project generator from templates
│   │   ├── ensure_idempotency.sh     #   Shared helpers (logging, env, checks)
│   │   ├── setup_ruby.sh             #   Ruby version verification
│   │   ├── setup_fastlane.sh         #   Conditional fastlane installation
│   │   ├── setup_match.sh            #   Certificate sync (readonly/init)
│   │   └── setup_secrets.sh          #   GitHub Secrets provisioning
│   ├── platform/
│   │   ├── fastlane/                 #   Generated fastlane configs
│   │   ├── templates/                #   Fastfile, Appfile, Matchfile, Xcode project templates
│   │   ├── signing/                  #   Signing artifacts
│   │   └── Gemfile                   #   Ruby dependencies
│   ├── policies/
│   │   ├── signing-policy.md         #   Certificate governance rules
│   │   └── release-policy.md         #   Release pipeline rules
│   └── config/
│       └── env.schema.json           #   Environment validation schema
│
├── .github/workflows/
│   ├── ci.yml                        #   Build & test on push/PR (no signing)
│   └── deploy.yml                    #   Signed build & TestFlight on v* tag
│
├── docs/                             #   Documentation
├── .env.platform.example             #   Organization-level config template
├── .env.app.example                  #   App-specific config template (committed)
└── .env.local.example                #   Developer override template (git-ignored)
```

### Separation of Concerns

| Directory | Contains | Depends on |
|-----------|----------|------------|
| `app/` | Product code, Xcode project, assets | Nothing outside `app/` |
| `release/` | Signing, fastlane, bootstrap, CI logic | Environment variables only |
| `.github/` | Workflow definitions | `release/` scripts, `.env.app`, and secrets |

fastlane, signing, CI, and bootstrap logic **never** exist inside `app/`.

## Environment Model

Configuration uses a three-layer system. Each layer overrides the previous:

```
.env.platform    (lowest — org secrets, git-ignored)
.env.app         (middle — app identity, committed)
.env.local       (highest — developer overrides, git-ignored)
```

`.env.app` contains no secrets and is committed to the repository so CI can read `APP_NAME` for builds.

### Required Variables

| Variable | Description | Example |
|----------|-------------|---------|
| `APPLE_TEAM_ID` | Apple Developer Team ID (10 chars) | `A1B2C3D4E5` |
| `MATCH_GIT_URL` | Signing repo URL (SSH recommended) | `git@github.com:org/certs.git` |
| `MATCH_PASSWORD` | Match encryption passphrase | *(secret)* |
| `APP_BUNDLE_ID` | Bundle identifier (reverse-DNS) | `com.company.myapp` |
| `APP_NAME` | Xcode project name (no spaces) | `MyApp` |
| `APP_DISPLAY_NAME` | Home screen name (spaces OK) | `My App` |

### Optional Variables

| Variable | Description | Default |
|----------|-------------|---------|
| `MATCH_TYPE` | Signing type | `appstore` |
| `ASC_KEY_ID` | App Store Connect API Key ID | — |
| `ASC_ISSUER_ID` | App Store Connect Issuer ID | — |
| `ASC_KEY_PATH` | Local path to `.p8` key file | — |
| `APPLE_ID` | Apple ID email used by fastlane/match for Apple Developer Portal access (recommended in .env.local) | — |
| `MATCH_GIT_PRIVATE_KEY_PATH` | Local path to SSH key for match repo | — |

`APP_SCHEME` and `XCODE_PROJECT_PATH` are automatically derived from `APP_NAME`.

## CI/CD

### CI — Every Push and PR (`ci.yml`)

Builds and tests without signing. Any contributor can verify the build passes.

### Deploy — Tagged Versions (`deploy.yml`)

```bash
git tag v1.2.0
git push origin v1.2.0
```

The pipeline:
1. Reads `APP_NAME` from committed `.env.app`
2. Sets up Ruby + bundler with caching
3. Runs `fastlane beta`: match sync → build → TestFlight upload

Build numbers are auto-incremented using `GITHUB_RUN_NUMBER + 1000` offset.

### Required GitHub Secrets

Provisioned automatically by `bootstrap.sh --init`:

| Secret | Source | Description |
|--------|--------|-------------|
| `MATCH_GIT_URL` | `.env.platform` | Signing repo URL |
| `MATCH_PASSWORD` | `.env.platform` | Match encryption passphrase |
| `MATCH_GIT_PRIVATE_KEY` | SSH key file | SSH access to signing repo |
| `APPLE_TEAM_ID` | `.env.platform` | Apple Developer Team ID |
| `APP_BUNDLE_ID` | `.env.app` | Bundle identifier |
| `ASC_KEY_ID` | `.env.app` | App Store Connect Key ID |
| `ASC_ISSUER_ID` | `.env.app` | App Store Connect Issuer ID |
| `ASC_KEY_CONTENT` | `.p8` file (base64) | App Store Connect private key |

## Bootstrap Modes

### Init Mode (`--init`)

Run once per repository:

```
bootstrap.sh --init
  ├── validate env against schema
  ├── generate Xcode project from templates (app/<APP_NAME>.xcodeproj)
  ├── verify Ruby >= 3.2 + fastlane
  ├── generate fastlane configs (Fastfile, Appfile)
  ├── verify signing repository exists
  ├── run match to generate certificates
  ├── provision GitHub Secrets (base64 ASC key, SSH match key)
  └── verify signing access (readonly sync)
```

### Normal Mode (default)

Validates without mutating:

```
bootstrap.sh
  ├── validate env against schema
  ├── ensure Xcode project exists
  ├── verify Ruby >= 3.2 + fastlane
  ├── verify GitHub Secrets exist
  └── run match in readonly mode
```

## Platform Contracts

### Idempotency

Every script checks existing state before acting. Running any command twice produces the same result.

### Non-Interactive Execution

No script prompts for input. Missing values produce a clear error and exit.

### Fail-Fast

Missing environment variables, unsupported OS, and schema violations cause immediate termination:

```
[FAIL] Required variable missing: APPLE_TEAM_ID
[FAIL] Environment validation failed — check .env files
```

### Certificate Safety

Destructive operations (`match nuke`, certificate deletion, key rotation) are **never** triggered automatically. See `release/policies/signing-policy.md`.

## Design Decisions

**Why generate the Xcode project instead of shipping a pre-built one?**
A pre-built project has placeholder values (bundle ID, team, display name) that require manual editing in Xcode's General tab. By generating the project from `.env` values, everything is correct from the first `open *.xcodeproj`.

**Why `APP_NAME` and `APP_DISPLAY_NAME` as separate variables?**
`APP_NAME` is used as a Swift identifier (`struct MyAppApp: App`), directory name, and Xcode target name — it cannot contain spaces. `APP_DISPLAY_NAME` is the user-facing name on the home screen — it can be anything.

**Why base64-encode the ASC key?**
GitHub Actions secrets are strings. A `.p8` file is binary-ish. The reference project that successfully deploys to TestFlight uses base64-encoded `ASC_KEY_CONTENT` with `is_key_content_base64: true` in fastlane. This is the proven pattern.

**Why SSH key for match instead of HTTPS?**
SSH key access to the signing repository is more reliable in CI environments and doesn't require storing git credentials. The reference project uses `MATCH_GIT_PRIVATE_KEY` for this purpose.
